Salesforce REST API Architecture

An enterprise-grade, extensible REST API framework built on Apex. Designed around a single abstract base class that handles HTTP routing, parameter parsing, logging, and error handling — so individual endpoints only need to implement business logic.

The architecture separates concerns into four layers: an HTTP layer (endpoint classes), a framework layer (BaseApi, RequestParams, ApiLogger, ErrorHandler), a service layer (business logic), and a data layer (fflib selectors). Cross-cutting concerns are inherited automatically by every endpoint that extends BaseApi.

Any exception thrown anywhere in the call chain bubbles up to the catch block in BaseApi.handleRequest(). The logging finally block always runs — even on errors — so every request is recorded regardless of outcome.

HTTP layer

WS_VGP_CitiesApi.cls

The only class with Salesforce REST annotations. It registers the /cities URL mapping and declares two static methods — one for GET, one for POST. Both do nothing but instantiate the class and call handleRequest(). All logic lives in the layers below.

Why two static methods calling the same thing?

Salesforce requires a separate static method per HTTP verb with the matching annotation (@HttpGet, @HttpPost). This is purely a platform constraint. The routing logic inside handleRequest() then reads req.httpMethod to decide which path to take.

@RestResource(urlMapping='/cities')
global with sharing class WS_VGP_CitiesApi extends BaseApi {

    global override Object handleGet(RequestParams params) {
        Integer page     = params.getInteger('page', 1);
        Integer pageSize = params.getInteger('page_size', 20);
        return ((ICitiesService) new CitiesService()).getCities(page, pageSize);
    }

    global override Object handlePost(RequestParams params) {
        List<Object> items = params.getList('cities');
        if (items.isEmpty()) throw new BadRequestException('cities array is required');
        return ((ICitiesService) new CitiesService()).createCities(items);
    }

    @HttpGet
    global static void getCities()     { new WS_VGP_CitiesApi().handleRequest(); }

    @HttpPost
    global static void createCities()  { new WS_VGP_CitiesApi().handleRequest(); }
}
Framework layer

BaseApi.cls

The single entry point for all requests. Owns the complete request lifecycle: timing, param parsing, HTTP method routing, response serialization, error catching, and guaranteed logging via finally. Subclasses only override handleGet() and/or handlePost().

global abstract class BaseApi {
    global void handleRequest() {
        Long startTime = System.currentTimeMillis();
        // ... builds logDto, creates RequestParams ...
        try {
            if      (req.httpMethod == 'GET')  response = handleGet(params);
            else if (req.httpMethod == 'POST') response = handlePost(params);
            else throw new BadRequestException('Unsupported method');
        } catch (Exception e) {
            // sets error status + body via ErrorHandler
        } finally {
            logDto.durationMs = System.currentTimeMillis() - startTime;
            ApiLogger.log(logDto);
            ApiLogger.flush(); // always runs
        }
    }
    global virtual Object handleGet(RequestParams p)  { throw new BadRequestException('GET not supported'); }
    global virtual Object handlePost(RequestParams p) { throw new BadRequestException('POST not supported'); }
}

RequestParams.cls

Abstracts parameter reading from both GET query strings and POST JSON bodies behind a single typed interface. Automatically throws BadRequestException for type mismatches (e.g. passing "abc" to getInteger()).

ApiLogger.cls

Buffers log records in a static list during the request, then inserts them all in a single DML call in the finally block. This keeps DML out of the hot path and ensures logging never counts against governor limits mid-request. Log insertion failures are caught and written to debug logs, so a logging failure never breaks the API response.

ErrorHandler.cls

A pure utility class. Takes any exception and returns the correct HTTP status code plus a structured error body. No state, no DML — just a mapping function.

Service layer

CitiesService.cls

Implements ICitiesService. Contains the two core business operations:

getCities(page, pageSize)

Paginates City__c records. Clamps page to minimum 1 and pageSize between 1 and 100. Runs two SOQL queries: one COUNT() for total records, one SELECT with LIMIT and OFFSET. Returns a ResponseWrapper containing the DTO list and a PaginationDTO.

createCities(items)

Validates each item individually first (name and governorate_sf_id required), then calls Database.insert(list, false) for partial success — meaning one invalid record does not roll back the others. Returns a BulkResponseWrapper showing the result per item index.

Data layer

CitySelector.cls

Extends fflib_SObjectSelector and implements ICitySelector. Uses the fflib Application factory (Application.Selector.newInstance()) for testability — the selector can be mocked in unit tests by registering a mock binding in the Application factory, avoiding real SOQL queries entirely during tests.

Known inconsistency: getSObjectFieldList() declares City_A__c but the manual SOQL queries use Name and Governorates__c. These should be aligned — either use the fflib query builder (newQueryFactory()) consistently, or add all queried fields to getSObjectFieldList(). Also, CitySelector is without sharing, while the service and API classes are with sharing — data queried here will not respect record visibility rules.
Exceptions

Exception hierarchy

All exceptions extend Salesforce's built-in Exception base class, which provides getMessage(), getTypeName(), and getStackTraceString() for free. The middle ApiException layer lets you catch all controlled API errors in one place while keeping unexpected system errors separate.

Salesforce built-in Exception    (platform — always available)
         ↑
    ApiException                 (your base — catch all API errors at once)
         ↑
  BadRequestException            (400 — bad input, wrong type, missing param)
  NotFoundException              (404 — record not found)
  ValidationException            (422 — field-level validation failure)
ClassHTTPWhen to throw
BadRequestException400Missing required param, wrong type, unsupported method, empty required array
NotFoundException404Queried record does not exist
ValidationException422Field-level validation — blank name, invalid foreign key, business rule violation
Anything else500Unhandled system exception — DML errors, null pointers, etc.

Adding a new endpoint

To add /governorates, you only need 3 new files. Logging, error handling, and param parsing are all inherited from BaseApi automatically.

// 1. Service interface
global interface IGovernoratesService {
    ResponseWrapper getGovernorates(Integer page, Integer pageSize);
}

// 2. Service implementation (same pattern as CitiesService)
global with sharing class GovernoratesService implements IGovernoratesService { ... }

// 3. Endpoint — inherits everything from BaseApi
@RestResource(urlMapping='/governorates')
global with sharing class WS_GovernoratesApi extends BaseApi {

    global override Object handleGet(RequestParams params) {
        Integer page     = params.getInteger('page', 1);
        Integer pageSize = params.getInteger('page_size', 20);
        return new GovernoratesService().getGovernorates(page, pageSize);
    }

    @HttpGet
    global static void getGovernorates() { new WS_GovernoratesApi().handleRequest(); }
}

Setup checklist

  • Create API_Log__c custom object with 8 fields: Endpoint__c, HTTP_Method__c, Request_Body__c, Status_Code__c, Response_Body__c, User__c (Lookup→User), Duration_Ms__c, Error_Message__c
  • Grant Create permission on API_Log__c to the integration user profile
  • Register City__c.SObjectType in the Application.Selector binding map (required for CitySelector.newInstance())
  • Deploy all 17 classes in the order listed in the Deploy Order section
  • Verify BulkResponseWrapper and BulkResultDTO exist in the org before deploying CitiesService
  • Ensure fflib library classes (fflib_SObjectSelector, fflib_ISObjectSelector) are installed
  • Assign the Remote Site Setting or Connected App permissions for external callers
